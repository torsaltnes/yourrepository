## REJECTION SUMMARY
- CSV export permits spreadsheet formula injection from user-controlled fields
- Attachment upload accepts unbounded base64 payloads into memory
- Some request fields lack null validation and can trigger 500s

Dependency audit was clean, so the rejection is implementation-level.

Main issue: `ExportToCsvAsync` only escapes commas, quotes, and newlines. It does not neutralize Excel/Sheets formula prefixes (`=`, `+`, `-`, `@`). Since `Title`, `ReportedBy`, `AssignedTo`, and `Tags` are attacker-controlled, a crafted deviation can become an active formula when the CSV is opened. That is a real injection risk.

Second issue: `AddAttachmentAsync` accepts arbitrary `Base64Content`, decodes it directly with `Convert.FromBase64String`, and stores the resulting bytes in memory with no size limit. In this in-memory app, that enables trivial memory exhaustion / DoS.

Additional gap: `UploadedBy` and `UpdatedBy` are not validated before `.Trim()`, so malformed requests can throw and surface as 500s instead of clean 400s. That’s not the primary blocker, but it confirms input validation is incomplete at the trust boundary.

I did not reject for missing auth; in this sandbox scope that is acceptable.

REJECTED
